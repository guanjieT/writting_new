# 修改清单

本文记录当前项目已经暴露的问题、已完成的修正，以及作为“上百万字超长篇小说编写 Agent”仍然缺失或不达标的能力。它不是逐行代码审查，而是面向系统能力、业务闭环和长期可维护性的改造清单。

## 本次已改正

### 1. 章节 Prompt fallback 退化为占位文本

**问题表现**

`FilePromptStore.DEFAULT_TEMPLATES["chapter"]` 之前只是：

```text
[请使用 prompts/chapter.md - 此文件仅为 fallback]
```

这会带来两个问题：

- 当 `prompts/chapter.md` 缺失时，章节生成 fallback 不包含 `当前章节计划`、`chapter_plan_focus_json`、`target_words_85` 等关键变量，无法形成可用的章节生成提示。
- 现有测试 `test_chapter_agent_uses_current_chapter_plan_as_parent` 会失败，因为测试故意使用不存在的 prompt 目录验证 fallback 行为。

**已修正**

已把 `chapter` fallback 改成可执行的最小章节生成模板，包含：

- 只输出 JSON 的约束。
- `content` 必须是完整正文的约束。
- 当前章节计划 `parent_artifact_json`。
- 章节计划结构化重点 `chapter_plan_focus_json`。
- 字数下限参考 `target_words_85`。

**验证**

已运行：

```bash
.venv/bin/pytest -q
```

结果：

```text
14 passed
```

### 2. Prompt 缺失会静默退回 fallback

**问题表现**

除 `chapter` 外，多个 fallback 仍主要是占位提示。正式运行时如果 `prompts/` 缺失或路径配置错误，系统可能启动成功，但生成质量会退化。

**已修正**

已新增启动自检：容器构建时默认检查所有 workflow step 和 `field_completion` 对应的 prompt 文件。缺失时直接抛出 `FileNotFoundError`，不再静默退回 fallback。

可通过 `NOVEL_AGENT_REQUIRE_PROMPT_FILES=false` 关闭该检查，主要用于测试或诊断。

### 3. 任务是内存态，不能支撑长篇生产

**问题表现**

`MemoryTaskStore` 只保存在进程内存，服务重启后任务状态丢失。

**已修正**

已新增 `FileTaskStore`，服务默认把任务写入 `data/tasks/*.json`：

- 任务创建、更新、查询、删除都走文件仓储。
- 服务重启后任务历史仍可查看。
- 启动时会把重启前仍处于 `pending` 或 `running` 的任务标记为 failed，并记录 interrupted 错误，避免前端无限等待。
- `MemoryTaskStore` 继续保留给测试和临时场景。

### 4. 项目 JSON 写入缺少原子性和并发保护

**问题表现**

项目 JSON 之前通过 `write_text()` 直接覆盖。多任务同时写同一项目时有覆盖风险，写入中断也可能留下半截 JSON。

**已修正**

`FileProjectRepository` 现在：

- 使用同目录临时文件写入。
- 写完后通过 `os.replace()` 原子替换目标 JSON。
- 在仓储实例内使用 `RLock` 保护 list/get/save/delete。

### 5. Artifact 没有版本管理

**问题表现**

同一作用域重新生成会覆盖当前 artifact，旧内容无法比较和回滚。

**已修正**

已为 artifact 注册流程增加版本历史：

- 新产物写入时自动附加 `version_id`、`previous_version_id`、`generated_at`。
- 如果同 key 已有 active artifact，会把旧 artifact 归档到 `project.artifact_history[key]`。
- 项目快照会返回 `artifact_history`，便于前端或后续工具查看历史版本。

### 6. 缺少长程知识沉淀

**问题表现**

生成链路只依赖当前上游摘要，没有一个项目级知识库沉淀稳定事实、约束和未决问题。

**已修正**

已新增轻量 `knowledge_base`：

- 每次注册 artifact 时，从 `summary.key_facts`、`summary.constraints`、`summary.open_questions` 抽取知识条目。
- 知识条目记录来源 artifact、步骤、类型和作用域。
- 生成上下文会带上相关知识库条目，Prompt 模板也已注入 `knowledge_base_json`。

### 7. 项目状态无法表达多卷多章进度

**问题表现**

`NovelProject.current_step/status` 是单值，不能表达多卷多章各自进展。

**已修正**

项目快照新增 `progress`：

- 统计目标卷数、每卷目标章节数、计划章节数、已完成正文数、完成比例。
- 按卷列出 `volume_outline`、`rough_chapter_plan` 状态。
- 按章列出 `chapter_plan`、`chapter`、`revision`、`consistency`、`memory` 状态。

### 8. Prompt 模板没有稳定注入真实上下文

**问题表现**

Prompt 文件内有 JSON 示例，如果使用 Python `str.format()` 会把示例大括号误判为变量。同时多个 Prompt 只列出变量说明，没有实际插入上下文。

**已修正**

- `PromptService.render()` 改为只替换 `{变量名}` 形式的安全渲染，JSON 示例不会破坏模板。
- 主要生成 Prompt 已加入“实际输入上下文”区块，注入项目简报、步骤输入、上游产物和知识库。

### 9. 缺少基础成稿导出

**问题表现**

项目有章节 artifact，但没有按卷章顺序汇出正文的接口。

**已修正**

新增 `GET /projects/{project_id}/exports/manuscript`：

- 按卷章顺序导出 Markdown 文本。
- 同一章节优先使用 `revision`，没有修订版时使用 `chapter` 草稿。
- 便于把项目从 artifact 工作台推进到成稿管理。

### 10. 缺少基础质量门禁

**问题表现**

之前只能看到任务和产物，不能快速判断章节是否缺正文、字数不足、缺一致性检查或缺记忆整理。

**已修正**

新增 `GET /projects/{project_id}/quality-report`：

- 按卷章检查章节计划、章节正文、修订、一致性检查、记忆整理。
- 标记 `missing_chapter_plan`、`missing_chapter`、`chapter_too_short`、`missing_consistency`、`missing_memory`。
- 返回 `ready_for_export` 和阻塞问题数量，作为基础质量门禁。

### 11. 后续章节上下文承接不足

**问题表现**

长篇生成时，后续章节不能只看当前章节计划，否则容易遗忘前面章节已经发生的事实、人物状态、物品归属、秘密暴露程度和未解决悬念。

**已修正**

- 生成上下文新增 `recent_context_json`，自动带入同卷最近前序章节的正文、修订、一致性检查和记忆摘要。
- 知识库筛选从“只看当前章”调整为“同卷当前章及之前章节”，避免后续章节看不到前文事实。
- `chapter_plan`、`chapter`、`consistency`、`memory` prompt 已明确要求读取前文上下文和知识库。

### 12. 前端学习成本偏高

**问题表现**

主工作流步骤多、依赖多，新用户需要自己判断下一步该点哪里。

**已修正**

- 主工作流侧栏新增“建议下一步”卡片，优先提示失败任务，其次提示当前上下文中已解锁但未生成的对象。
- 审查台摘要接入质量报告和进度矩阵，直接展示正文完成数、质量阻塞数和是否可导出。
- 主工作流页面改为“创作驾驶舱”表达，左侧新增阶段进度条，用户可以先看设定、结构、写作、审查各阶段完成度，再进入具体对象。
- 当前工作区新增“读取 / 产出 / 范围 / 影响”四格说明，帮助用户理解每个按钮会读哪些上游产物、写入哪个卷章范围，以及会影响哪些下游步骤。
- 执行按钮从泛化的“执行当前对象”改为“生成/重新生成 + 对象名”，降低第一次使用时的认知负担。

## 当前仍存在的问题

### 1. 章节正文生成只有一次扩写补救

`ChapterAgent` 已经有长度检查和一次扩写补救，这是好方向，但仍不够支撑长篇生产。

**影响**

- 如果模型反复短写，只能失败。
- 没有分 scene 生成和拼接策略。
- 没有“先生成场景，再整体润色”的稳定长文流水线。

**建议**

- 把章节正文拆成 scene 级生成。
- 每个 scene 单独校验目标、冲突、人物、信息增量。
- 最后合并、去重、统一语气，再做章节级审查。

### 2. 知识库仍是轻量沉淀，不是可检索长程记忆系统

目前已经有轻量知识沉淀和前序章节上下文，但还不是全局可检索知识库。

**影响**

- 百万字小说需要持续记住人物状态、地理位置、时间线、伏笔、能力边界、组织关系。
- 只靠上游 artifact 摘要无法覆盖数百章的细节。

**建议**

- 建立结构化知识库：角色、地点、势力、物品、能力、事件、伏笔、未决问题。
- 每章生成后自动抽取并更新知识库。
- 生成新章节前按卷、章、人物、地点、事件检索相关记忆。

### 3. 一致性检查仍是单步报告，不是强约束系统

当前 `consistency` 能生成报告，但不会阻止错误进入后续章节，也不会自动回写修复任务。

**影响**

- 长篇中设定漂移会积累。
- 冲突发现后缺少闭环处理。

**建议**

- 生成前做硬约束读取。
- 生成后做自动一致性审查。
- 审查失败时自动创建修订任务或阻止进入下一章。
- 对严重冲突提供“修正文案”和“影响范围”。

### 4. 缺少全书级叙事调度

当前工作流有总纲、卷纲、章节计划，但缺少跨百万字的宏观调度器。

**缺口**

- 伏笔投放和回收表。
- 角色弧光进度表。
- 主线/支线比例控制。
- 爽点、低谷、反转、信息揭示的密度控制。
- 卷间节奏和章间节奏的长期监控。

**建议**

- 增加 `narrative_scheduler` 或类似模块。
- 每卷、每十章、每五十章做节奏审查。
- 对未来章节计划提供“必须推进/暂缓推进/禁止提前揭示”的调度建议。

### 5. 基础质量门禁已具备，但缺少模型级质量评估和多轮修订流水线

现在有 `revision` 和确定性质量报告，但还没有模型级审稿、多维评分和自动多轮修订。

**长篇需求**

一章正文至少需要多维评估：

- 是否完成章节计划。
- 是否符合人物动机。
- 是否有真实场景推进。
- 是否有节奏问题。
- 是否有重复信息。
- 是否存在设定冲突。
- 是否达到字数和可读性要求。

**建议**

- 增加自动评分器。
- 把评分结果转成 revision instructions。
- 支持多轮修订直到达到阈值或人工确认。

### 6. 成稿导出已有基础接口，但缺少定稿管理

当前已有基础 Markdown 成稿导出，但仍不适合完整百万字成稿管理。

**缺口**

- 保留修订版和定稿版。
- 统计字数、章节数、完成率。
- 生成目录、卷标题、章节标题。

**建议**

- 将 `chapter` 和 `revision` 区分为草稿版、修订版、定稿版。
- 前端增加成稿视图。

### 7. 缺少模型成本、上下文预算和失败恢复策略

百万字写作会消耗大量 token 和调用次数。

**缺口**

- 没有 token 预算管理。
- 没有每章成本估算。
- 没有失败自动降级策略。
- 没有模型能力分层，例如规划用便宜模型、正文用强模型、审查用不同模型。

**建议**

- 给每个步骤配置模型、温度、max_tokens、重试策略。
- 记录每次调用 token 和成本。
- 对 JSON 解析失败、长度不足、超时分别设置不同重试策略。

## 百万字长篇 Agent 的能力缺口

### 必须补齐的核心能力

1. **全书知识库**
   - 角色状态、地点状态、时间线、势力关系、能力体系、物品归属、伏笔、未决问题都应结构化保存。

2. **检索增强上下文**
   - 生成某章前不能只读当前章节计划，还要按人物、地点、事件和伏笔检索相关历史。

3. **场景级生成**
   - 百万字不能依赖一次 prompt 生成整章。应从章节计划拆到 scene，再 scene 生成、scene 校验、章节合并。

4. **连续性约束系统**
   - 不是生成后才问有没有冲突，而是在生成前把硬约束注入，在生成后强制检查。

5. **版本与回滚**
   - 长篇迭代一定会改设定、改章节。没有版本系统就无法安全试错。

6. **进度矩阵**
   - 需要全书/卷/章/场景级状态，而不是项目单一 status。

7. **质量门禁**
   - 每章要有自动评分、错误报告、修订建议和是否允许进入下一章的判断。

8. **批量调度**
   - 支持按卷批量生成粗章纲、按章节队列生成正文、失败重试、暂停恢复。

9. **成稿出口**
   - 最终目标是成书，不只是 artifact 展示。必须有导出、合并、定稿和字数统计。

10. **人工确认节点**
    - 长篇关键节点需要“锁定”：总纲锁定、卷纲锁定、角色设定锁定、章节定稿锁定。

### 当前已有但需要增强的能力

| 能力 | 当前状态 | 改进方向 |
| --- | --- | --- |
| 项目中心 | 已有基础字段 | 增加作品 bible、读者定位、商业类型、禁忌强度 |
| 总纲/卷纲/章节计划 | 已有五段式规划 | 增加节奏调度、伏笔表、角色弧光表 |
| 章节正文 | 已能基于章节计划生成 | 改为 scene 级生成与合并 |
| 修订 | 已有单步入口 | 增加多轮修订和质量评分闭环 |
| 一致性检查 | 已有报告入口 | 改为生成前硬约束和生成后门禁 |
| 记忆整理 | 已有步骤 | 改为持续更新的知识库与检索系统 |
| 审计 | 已有 JSONL | 增加版本、任务输入、模型调用和成本审计 |
| 前端工作台 | 已能操作主流程 | 增加进度矩阵、知识库视图、批量任务视图 |

## 建议优先级

### P0：先保证流程可靠

- 更细粒度的项目写锁或数据库级并发控制。
- 任务输入 payload、模型配置、重试次数、耗时和错误栈的完整审计。

### P1：让它能写长篇而不丢设定

- 建立知识库。
- 每章后自动抽取记忆。
- 生成前检索相关记忆。
- 增加一致性门禁。

### P2：让章节质量稳定

- scene 级生成。
- 自动评分。
- 多轮修订。
- 章节定稿状态。

### P3：让百万字项目可管理

- 卷章进度矩阵。
- 批量任务队列。
- 成稿导出。
- 成本统计。
- 人工确认和锁定机制。

## 后续修改原则

- 不再新增只靠正文正则判断结构的逻辑，卷章判断必须依赖 structured。
- 不把所有能力塞进单个大文件；知识库、调度、导出、质量评估都应拆成独立模块。
- 所有新增前端字段必须同步后端 schema、prompt 和文档。
- 所有会影响下游的改动都要有 stale、版本或锁定策略。
- 所有长篇核心能力都要能审计：谁生成、基于什么输入、用哪个模型、产生哪个版本。
