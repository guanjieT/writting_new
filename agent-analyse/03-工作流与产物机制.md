# 工作流与产物机制

## 工作流定义

`workflow_spec.py` 是工作流的唯一权威定义。每个步骤包含：

- `key`：后端和前端共同使用的步骤标识。
- `label`：展示名称。
- `depends_on`：前置依赖。
- `page`：主要归属页面。
- `scope`：产物作用域，分为 `project`、`volume`、`chapter`。

当前链路是：

```text
requirements
  -> story_bible
  -> characters
  -> outline
  -> rough_volume_outline
  -> volume_outline
  -> rough_chapter_plan
  -> chapter_plan
  -> chapter
  -> revision / consistency / memory
```

前端也有一份 fallback 步骤定义，用于 `/workflow/spec` 不可用时兜底，但正常情况下后端 `workflow_spec()` 才是主来源。

## 作用域设计

小说生成不是只有全局产物。卷纲、章节计划、正文都需要区分“第几卷、第几章”。因此项目里引入了作用域。

| 作用域 | 示例步骤 | artifact key 示例 | metadata |
| --- | --- | --- | --- |
| project | `outline` | `outline` | `scope_kind=project` |
| volume | `volume_outline` | `volume_outline:1` | `scope_kind=volume`, `volume_index=1` |
| chapter | `chapter` | `chapter:1:2` | `scope_kind=chapter`, `volume_index=1`, `chapter_index=2` |

后端通过 `_artifact_scope_payload()` 从 payload 推导当前作用域，通过 `_artifact_key()` 生成稳定 key，通过 `_scope_matches()` 判断某个 artifact 是否是当前步骤需要的那个。

这个机制避免了一个常见问题：如果所有章节正文都叫 `chapter`，后写的章节会覆盖先写的章节。现在每卷每章都有独立 key。

## 依赖校验

每次执行生成前，`NovelOrchestrator._ensure_step_ready()` 会检查：

1. 所有 `depends_on` 中声明的依赖是否存在。
2. 依赖 artifact 是否被标记为 stale。
3. 如果是 `volume_outline`，还会检查粗卷纲 structured 里是否存在目标卷。
4. 如果是 `chapter_plan`，还会检查粗章纲 structured 里是否存在目标章。

这里的关键点是：校验不靠正文里有没有“第2卷”“第3章”这样的文本，而是看 structured 字段。这比正则扫正文可靠得多。

## stale 机制

上游重新生成后，下游旧结果可能不再可信。项目通过 stale 标记处理这个问题。

例如重新生成第 1 卷的 `volume_outline:1`：

- 依赖它的 `rough_chapter_plan:1` 会 stale。
- 第 1 卷下已有 `chapter_plan:1:x`、`chapter:1:x`、`revision:1:x` 等也会 stale。
- 第 2 卷的产物不受影响。

删除 artifact 时走另一个策略：不是标记 stale，而是直接删除目标 artifact 和它作用域内的后代 artifact。这样用户明确删除上游产物时，后续链路会回到未生成状态。

## 结构化输出

Agent 要求 LLM 返回 JSON，而不是自由文本。通用结构是：

```json
{
  "content": "...",
  "overview": "...",
  "key_facts": [],
  "constraints": [],
  "open_questions": [],
  "best_for_reason": "...",
  "structured": {}
}
```

其中：

- `content` 是可读正文或摘要。
- `overview`、`key_facts`、`constraints` 等用于下游快速继承。
- `structured` 是规划类产物真正的主体数据。

部分步骤有强校验：

| 步骤 | 必须包含的 structured |
| --- | --- |
| `outline` | `story_arcs`、`global_goals`、`main_conflicts`、`ending_direction`、`volume_plan_hints` |
| `rough_volume_outline` | `volumes` |
| `volume_outline` | `volume_index`、`title`、`summary`、`goal`、`main_conflict`、`ending_hook`、`target_chapter_count`、`target_words`、`arc_segments` |
| `rough_chapter_plan` | `volume_index`、`total_target_chapters`、`chapters` |
| `chapter_plan` | `volume_index`、`chapter_index`、`title`、`summary`、`chapter_type`、`pov_character`、`main_event`、`conflict`、`hook`、`target_words`、`min_words` 等 |

粗卷纲会校验卷序号连续，粗章纲会校验章节数与 `total_target_chapters` 一致，并且章节序号连续。

## Artifact 生命周期

一个 artifact 从生成到下游使用大致经历：

```text
LLM JSON
  -> GeneratedArtifactPayload
  -> ArtifactSummary
  -> Artifact
  -> project.artifacts[key]
  -> snapshot API
  -> 前端展示 / 下游上下文选择
```

artifact 的 `metadata` 是工作流正确性的关键，里面记录：

- `step`
- `scope_kind`
- `volume_index`
- `chapter_index`
- `state`
- `stale`
- `agent`
- `generated_payload`

下游步骤优先读取 `summary.structured`，必要时再读取压缩后的正文节选。这样既能保留完整产物用于审查，又不会每次都把全文注入 prompt。

## 任务与工作流的关系

路由不会直接生成 artifact，而是创建任务。任务记录有自己的作用域字段，方便前端区分“第 1 卷第 2 章的 chapter 任务”和其他任务。

任务完成后，结果里会包含 AgentOutput 的 JSON；但项目真正可持续使用的数据在 `project.artifacts` 里。

换句话说：

- `TaskRecord` 是执行历史。
- `Artifact` 是项目状态。
- `AuditEvent` 是可追踪操作日志。
