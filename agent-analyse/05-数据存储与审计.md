# 数据存储与审计

## 项目数据

项目数据保存在 `data/projects/*.json`。每个文件对应一个 `NovelProject`，文件名是项目 ID。

项目 JSON 里最重要的字段是：

- `project_id`
- `input`
- `current_step`
- `status`
- `artifacts`
- `artifact_history`
- `knowledge_base`
- `created_at`
- `updated_at`

`FileProjectRepository` 负责读写这些 JSON。它会校验 project_id 只能由安全字符组成，避免路径穿越。项目保存现在使用同目录临时文件加原子替换，并在仓储实例内加锁，降低并发写入和中断写入导致半截 JSON 的风险。

## Artifact 存储形态

artifact 嵌在项目 JSON 的 `artifacts` 字典中，不是单独文件。

一个 artifact 通常包含：

```json
{
  "key": "chapter_plan:1:2",
  "title": "章节计划",
  "content": "...",
  "summary": {
    "summary": "...",
    "overview": "...",
    "key_facts": [],
    "constraints": [],
    "open_questions": [],
    "best_for_steps": [],
    "best_for_reason": "...",
    "structured": {}
  },
  "metadata": {
    "agent": "...",
    "step": "chapter_plan",
    "scope_kind": "chapter",
    "volume_index": 1,
    "chapter_index": 2,
    "stale": false,
    "state": "active",
    "generated_payload": {}
  }
}
```

这里要注意两个层次：

- `content` 是人类可读内容。
- `summary.structured` 是下游逻辑更依赖的机器可读结构。

`metadata.generated_payload` 保存模型原始结构化 payload 的副本，方便审查“模型到底返回了什么”。

## Artifact 版本历史

同一 artifact key 重新生成时，当前 active artifact 会被归档到 `artifact_history[key]`，新 artifact 会继续写在 `artifacts[key]`。新 artifact 的 metadata 会包含：

- `version_id`
- `previous_version_id`
- `generated_at`

这让长篇项目可以保留设定、卷纲、章节计划和正文的历史版本，为后续 diff、回滚和审计打基础。

## 知识库

项目会把每次生成产物的 `key_facts`、`constraints`、`open_questions` 沉淀到 `knowledge_base`。每条知识记录会包含来源 artifact、来源步骤、知识类型和作用域。

生成上下文会携带相关知识库条目，Prompt 模板也会读取 `knowledge_base_json`。当前实现是轻量沉淀，还不是向量检索或复杂知识图谱。

章节级生成还会携带 `recent_context_json`：同卷最近前序章节的正文、修订、一致性检查和记忆摘要。知识库筛选会包含同卷当前章及之前章节的知识条目，避免后续章节丢失前文状态。

## 任务存储

服务默认使用 `FileTaskStore`，任务保存在 `data/tasks/*.json`。

这意味着：

- 服务运行期间可以查看任务状态。
- 服务重启后任务历史仍可查看。
- 重启前仍处于 `pending` 或 `running` 的任务会被标记为 failed，并记录 interrupted 错误，避免前端无限等待。
- 已经写入项目 JSON 的 artifacts 不会丢。

`MemoryTaskStore` 仍保留给测试和临时场景。后续如果要做多用户或长时间生产环境，可以继续把 `TaskStore` 换成 SQLite、PostgreSQL 或队列系统。

## 审计日志

审计日志写入 `data/audit/events.jsonl`。每一行是一个 `AuditEvent` JSON。

常见审计动作包括：

- `project.created`
- `project.updated`
- `project.deleted`
- `project.artifact.deleted`
- `agent.{step}.completed`
- `project.field_completed`

审计日志是旁路系统，不参与工作流校验。它的价值是追踪“什么时候、哪个项目、执行了什么动作、产生了什么 artifact”。

## Prompt 存储

Prompt 模板保存在 `prompts/*.md`。`FilePromptStore` 根据模板名读取对应文件。

服务默认会在启动时检查所有 workflow step 和 `field_completion` 对应的 prompt 文件是否存在。缺失时直接报错，避免正式生成静默退回 fallback。`NOVEL_AGENT_REQUIRE_PROMPT_FILES=false` 时才允许读取内置 fallback，这主要用于测试或诊断。

实际生成质量主要由这些 Prompt 文件决定；后端代码更关注流程、安全校验和结构化输出解析。

## LLM Provider

项目有两个 provider：

| Provider | 用途 |
| --- | --- |
| `MockLLMProvider` | 本地跑通流程，返回占位 JSON 或占位草稿 |
| `OpenAICompatibleProvider` | 调用兼容 OpenAI `/chat/completions` 的模型服务 |

Mock Provider 不追求写作质量，它的作用是让流程、前端、依赖校验、artifact 落盘可以离线验证。

OpenAI-compatible Provider 用标准 HTTP 请求发送：

- system prompt
- user prompt
- model
- temperature
- max_tokens

返回后只取 `choices[0].message.content`，再交给 Agent 做 JSON 解析。

## 配置路径

配置优先从环境变量读取，常用变量包括：

- `NOVEL_AGENT_DATA_DIR`
- `NOVEL_AGENT_PROMPTS_DIR`
- `NOVEL_AGENT_PROJECTS_DIR`
- `NOVEL_AGENT_TASKS_DIR`
- `NOVEL_AGENT_AUDIT_LOG`
- `NOVEL_AGENT_REQUIRE_PROMPT_FILES`
- `NOVEL_AGENT_LLM_PROVIDER`
- `NOVEL_AGENT_LLM_API_KEY`
- `NOVEL_AGENT_LLM_BASE_URL`
- `NOVEL_AGENT_LLM_MODEL`

如果没有配置真实模型，系统默认使用 Mock Provider。因此“服务能启动”和“真实模型能生成高质量内容”是两件不同的事。

## 数据一致性重点

维护这个项目时，最需要保护的是以下一致性：

- artifact key 与 metadata 作用域一致。
- 前端选择的卷章与后端 payload 中的卷章一致。
- structured 中的卷号、章号与 metadata 中的卷号、章号一致。
- 上游改变后，下游 stale 或删除逻辑正确。
- 项目更新后旧任务和旧产物被清空。

这些点一旦失守，就容易出现“前端看到的是第 2 章，后端用的是第 1 章计划”之类的隐蔽错误。
