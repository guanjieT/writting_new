# 扩展与维护指南

## 新增一个工作流步骤

新增步骤时，不要只加一个路由或一个按钮。这个项目的步骤横跨后端、Prompt、前端和依赖校验。

建议顺序：

1. 在 `workflow_spec.py` 添加 `WorkflowStepSpec`，明确 key、label、depends_on、page、scope。
2. 在 `domain.WorkflowStep` 添加枚举值。
3. 在 `api/schemas.py` 添加请求模型。
4. 在 `agents.py` 添加 Agent 类，并加入 `build_agent_registry()`。
5. 在 `prompts/` 添加对应模板。
6. 在 API route 中添加提交接口，或复用已有模式。
7. 在 `frontend/assets/js/step-meta.js` 添加字段、endpoint 和展示信息。
8. 如果步骤有特殊作用域或父级产物，检查 `workspace-utils.js` 的 `buildBaseArtifactPayload()`。
9. 如果下游依赖有特殊结构校验，补到 `orchestrator._ensure_step_ready()` 或 Agent 的 structured 校验中。
10. 添加或更新测试。

## 修改字段

修改表单字段时要沿着数据链路查：

```text
step-meta.js 字段
  -> workspace-utils.js serializeStepPayload
  -> api-client.js endpoint
  -> api/schemas.py 请求模型
  -> context_builder.py step_payload
  -> prompt 模板变量
  -> Agent 输出校验
```

如果字段是项目中心字段，还要检查：

```text
index-page.js
  -> ProjectInput
  -> 前端默认值/回填逻辑
  -> 字段补全上下文
  -> 相关 prompt
```

最常见的问题不是字段没传，而是字段传到了后端，却没有进入 Prompt 或没有被 structured 校验覆盖。

## 修改作用域逻辑

作用域相关代码集中在：

- `workflow_spec.py` 的 `scope`。
- `services.py` 的 `_artifact_scope_payload()`、`_artifact_key()`、`_scope_matches()`。
- `orchestrator.py` 的依赖校验。
- `frontend/assets/js/scope-utils.js`。
- `frontend/assets/js/artifact-selectors.js`。
- `frontend/assets/js/planning-defaults.js`。

改这里要特别小心，因为它决定“当前页面操作的是哪个卷、哪个章”。最好同时补测试，覆盖至少两个卷、每卷两个章的情况。

## 接入真实模型

真实模型接入主要改环境变量，不需要改业务代码：

```text
NOVEL_AGENT_LLM_PROVIDER=openai-compatible
NOVEL_AGENT_LLM_API_KEY=...
NOVEL_AGENT_LLM_BASE_URL=...
NOVEL_AGENT_LLM_MODEL=...
```

如果模型返回不是 JSON，错误会在 Agent 解析阶段暴露。排查顺序：

1. 看任务错误。
2. 看模型原始返回是否被代码块包裹或夹杂解释文字。
3. 看 prompt 是否明确要求只输出 JSON。
4. 看 structured 必填字段是否满足。
5. 看 `max_tokens` 是否太小导致 JSON 被截断。

## 调整 Prompt

Prompt 模板是生成质量的主要控制面。

调整时建议遵守：

- 保持 JSON contract 明确。
- 不要让模型自由改变字段名。
- 对规划类步骤明确 structured 必填字段。
- 对正文步骤明确正文长度和禁止用总结凑字。
- 对下游步骤说明应优先继承上游 structured，而不是自由重构。

如果修改了输出字段，必须同步更新 `agents.py` 的校验和前端展示/解析逻辑。

## 排查生成失败

生成失败一般分几类：

| 现象 | 可能原因 | 重点查看 |
| --- | --- | --- |
| 接口返回依赖缺失 | 前置步骤没生成或作用域不匹配 | `project.artifacts`、metadata、payload |
| 接口返回 stale | 上游重新生成后下游未重跑 | artifact metadata |
| 任务 failed 且 JSON 解析错误 | 模型没有按 JSON 输出 | prompt、模型返回、max_tokens |
| 章节计划找不到目标章 | 粗章纲 structured 没有该章 | `rough_chapter_plan.summary.structured.chapters` |
| 卷纲找不到目标卷 | 粗卷纲 structured 没有该卷 | `rough_volume_outline.summary.structured.volumes` |
| 前端按钮不可点 | 前端依赖判断认为缺失或 stale | `artifact-selectors.js`、当前 URL 卷章参数 |
| 前端回填和后端不一致 | 表单序列化或 hidden select 问题 | `workspace-utils.js`、`scope-utils.js` |

## 维护建议

- 保持 `README.md`、`docs/parameter-flow-audit.md` 和本目录文档同步。
- 新增字段时优先查现有工具函数，避免重复写序列化或作用域判断。
- 后端安全校验不能只依赖前端 UI 状态。
- 规划类产物继续以 structured 为主，不要退回从正文里正则提取卷章。
- 大改 Prompt 后至少用 Mock 跑通全链路，再用真实模型小样本验证 JSON 稳定性。
