# 后端核心逻辑

## 领域模型

`domain.py` 定义了系统的基础语言。

最重要的模型有：

| 模型 | 含义 |
| --- | --- |
| `ProjectInput` | 项目中心填写的长期设定，如标题、题材、前提、字数、卷数、风格和禁忌 |
| `NovelProject` | 一个完整项目，包含输入设定、当前状态、artifact 字典和时间戳 |
| `Artifact` | 某一步生成出来的产物，包含正文、结构化摘要和 metadata |
| `ArtifactSummary` | 面向下游步骤读取的摘要结构，如 overview、key_facts、constraints、structured |
| `AgentContext` | 单次 Agent 执行时的运行参数和请求 payload |
| `AgentOutput` | Agent 返回的步骤结果，主体是一个 artifact |
| `TaskRecord` | 异步任务记录，包括状态、结果、错误和作用域 |
| `AuditEvent` | 审计事件，记录项目创建、生成完成、字段补全等动作 |

`domain.py` 还定义了多个协议接口，例如 `ProjectRepository`、`PromptStore`、`LLMProvider`、`TaskStore`。这些协议让服务层依赖抽象而不是依赖具体文件实现。

## 配置与容器

`config.py` 负责把环境变量整理成 `AppConfig`。它处理数据目录、Prompt 目录、项目目录、审计日志路径、服务端口、CORS、LLM 配置等。

`container.py` 是依赖注入中心。它把以下对象组装到一起：

- `FileProjectRepository`
- `FilePromptStore`
- `FileAuditSink`
- `FileTaskStore`
- `MockLLMProvider` 或 `OpenAICompatibleProvider`
- `PromptService`
- `LLMService`
- `ProjectService`
- `AuditService`
- `TaskService`
- `NovelOrchestrator`

`build_llm_provider()` 的策略很直接：如果配置里声明了 `openai-compatible`、`openai` 或 `deepseek`，并且 key/base_url/model 都存在，就走真实接口；否则退回 Mock Provider。

## 服务层

`services.py` 是后端状态管理的核心。

### ProjectService

`ProjectService` 负责项目生命周期：

- 创建项目。
- 获取项目。
- 列出项目。
- 保存项目。
- 删除项目。
- 注册生成产物。
- 删除某个产物并清理依赖它的后续产物。
- 归档同 key 旧产物，维护 artifact 历史版本。
- 从 artifact 摘要中沉淀项目级知识库条目。

它还负责 stale 传播。一个上游 artifact 被重新生成时，后续依赖它的 artifact 会被标记为 `stale`。这可以防止用户拿旧章节计划继续生成正文。

重新生成同一 artifact key 时，旧 active artifact 会进入 `project.artifact_history`，新 artifact 会获得新的 `version_id`，并记录 `previous_version_id`。

### TaskService

`TaskService` 把生成步骤包装成后台任务。路由提交任务后立即返回 `TaskRecord`，真实生成逻辑在线程池里跑。

任务状态流转大致是：

```text
pending -> running -> completed
pending -> running -> failed
pending -> cancelled
```

服务默认使用 `FileTaskStore`，任务记录会写入 `data/tasks/*.json`。服务重启后可继续查看历史任务；如果重启前任务还处于 `pending` 或 `running`，启动时会标记为 failed，避免前端一直显示执行中。`MemoryTaskStore` 仍保留给测试和临时场景使用。

### PromptService 与 LLMService

`PromptService` 从 `prompts/*.md` 读取模板，并用 `str.format(**variables)` 渲染变量。

`LLMService` 是薄封装，只负责把请求转给 provider。具体 provider 可以是 Mock，也可以是 OpenAI-compatible HTTP 接口。

### AuditService

`AuditService` 负责写审计日志。如果配置关闭审计，它就什么也不做。审计并不影响主流程，是旁路记录。

## 编排器

`orchestrator.py` 是“业务控制台”。

`dispatch(project_id, step, payload)` 的职责是：

1. 读取项目。
2. 确认 step 是否支持。
3. 检查前置依赖是否存在。
4. 检查依赖是否 stale。
5. 对卷纲、章节计划做额外结构化检查。
6. 构造 `AgentContext`。
7. 调用对应 Agent。
8. 把生成 artifact 注册回项目。
9. 写审计日志。

这里有一个重要边界：编排器不拼 prompt，也不直接操作底层文件。它只用服务层和 Agent 完成业务动作。

## Agent 层

`agents.py` 里所有具体 Agent 都继承自 `BaseAgent`，共同逻辑包括：

- 根据当前步骤构造上下文变量。
- 读取并渲染 prompt。
- 调用 LLM。
- 去掉代码块包裹。
- 解析 JSON。
- 校验 structured 字段。
- 生成 `ArtifactSummary`。
- 生成带 metadata 的 `Artifact`。

规划类 Agent 继承 `PlanningAgent`，会额外寻找父级 artifact，例如：

- `volume_outline` 读取 `rough_volume_outline`。
- `chapter_plan` 读取 `rough_chapter_plan`。
- `chapter` 读取 `chapter_plan`。

`ChapterAgent` 有额外字数保护：如果生成正文长度明显不足，会自动构造扩写 prompt 进行一次补救；补救后仍不足，就返回明确错误，提示提高 `max_tokens`。

Agent 共用的上下文辅助函数拆在 `agent_helpers.py`，包括总纲目标参数构造、父级 artifact prompt 上下文压缩、章节计划重点字段抽取和文本列表归一化，避免 `agents.py` 继续膨胀。

## 上下文构造

`context_builder.py` 负责把项目、当前 payload、上游 artifact 压成适合 prompt 使用的上下文。

它不是把整个项目和所有全文都塞给模型，而是按阶段选择不同上下文档位：

- 总纲阶段读项目简报和上游核心摘要。
- 卷纲阶段读总纲和粗卷纲相关结构。
- 章节计划阶段读卷纲、粗章纲和相关设定。
- 章节正文阶段重点读当前章节计划。
- 修订、一致性、记忆阶段读当前章节和相关审查材料。

这种做法能控制 prompt 体积，也能降低下游步骤被无关旧内容干扰的概率。
