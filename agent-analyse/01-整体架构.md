# 整体架构

## 项目定位

`novel_agent_clean` 是一个面向长篇小说生产的 Agent 工作流框架。它解决的不是“单次让模型写一段文本”，而是把小说生成拆成可审查、可回滚、可继续扩展的一串阶段：

```text
项目设定
  -> 需求分析
  -> 世界观设定
  -> 角色框架
  -> 总纲
  -> 粗卷纲
  -> 完整卷纲
  -> 粗章纲
  -> 完整章节计划
  -> 章节正文
  -> 修订 / 一致性检查 / 记忆整理
```

系统的中心对象是 `NovelProject`。所有生成结果都挂在项目的 `artifacts` 字典里，所有接口、页面、任务和审计日志最终都围绕项目 ID 运转。

## 分层结构

后端代码主要分成六层：

| 层级 | 代表文件 | 职责 |
| --- | --- | --- |
| 领域层 | `domain.py` | 定义项目、产物、任务、工作流状态、接口协议和业务异常 |
| 工作流定义 | `workflow_spec.py` | 统一声明步骤顺序、依赖、页面归属和作用域 |
| 服务层 | `services.py` | 管理项目、任务、审计、Prompt、LLM 调用和 artifact 工具函数 |
| Agent 层 | `agents.py` | 根据项目和上下文渲染 prompt、调用模型、解析并校验输出 |
| 编排层 | `orchestrator.py` | 检查依赖、分派步骤、注册产物、记录审计 |
| GUI 层 | `gui/` | tkinter 桌面界面：视图、表单、产物展示、任务轮询 |

基础设施在 `infrastructure.py` 中实现，包括文件项目仓储、Prompt 存储、审计写入、文件任务存储、测试用内存任务存储、Mock LLM 和 OpenAI-compatible Provider。

GUI 代码在 `novel_agent/gui/`：

| 文件/目录 | 职责 |
| --- | --- |
| `app.py` | Tk 根窗口、DI 容器初始化、侧栏导航、视图切换 |
| `state.py` | 共享状态数据类 |
| `theme.py` | GUI 统一视觉主题、ttk 样式和原生 Text/Listbox 样式工具 |
| `step_fields.py` | 12 个步骤的字段定义（Python 版 step-meta.js） |
| `scope_utils.py` | 范围解析、产物查找、结构化数据解析、依赖检查 |
| `task_monitor.py` | tkinter.after() 异步任务轮询 |
| `widgets/form_builder.py` | 动态表单：FieldDef → tkinter 控件 |
| `widgets/artifact_viewer.py` | 产物只读展示 |
| `views/project_view.py` | 项目中心：创建、列出、编辑、删除 |
| `views/workflow_view.py` | 主工作台：步骤导航、表单、产物查看 |
| `views/review_view.py` | 审查面板：统计、进度矩阵、任务历史、质量报告 |

## 运行入口

GUI 模式：`python -m novel_agent.gui`

入口在 `gui/__main__.py`，创建 Tk 窗口后通过 `App` 类初始化。`App.__init__()` 中：

1. 调用 `load_config()` 读取环境变量。
2. 调用 `build_container()` 组装仓储、服务、Agent 编排器和任务线程池。
3. 调用 `theme.apply_theme()` 初始化统一视觉风格。
4. 创建侧栏（项目信息、导航、范围选择）和三个视图。
5. GUI 直接通过 `container.orchestrator`、`container.task_service` 等调用后端。

## 核心对象关系

```
GUI 事件
  -> TaskService.submit(...)
    -> NovelOrchestrator.dispatch(...)
      -> ProjectService.get(...)
      -> _ensure_step_ready(...)
      -> Agent.run(...)
        -> build_generation_context(...)
        -> PromptService.render(...)
        -> LLMService.complete(...)
        -> parse generated JSON
      -> ProjectService.register_generated_artifact(...)
      -> AuditService.record(...)
```

用户看到的是”点击生成按钮”，系统内部实际发生的是”创建任务 -> 后台执行 -> 调 Agent -> 产物落盘 -> GUI 轮询任务状态 -> 刷新快照”。

## 设计思路

这个项目的主要设计目标是解耦：

- GUI 层只负责界面和交互，不承担生成逻辑。
- GUI 视觉样式集中在 `theme.py`，视图只复用样式名和少量原生控件样式工具，避免颜色、字体、间距散落在各页面。
- 服务层只管理状态和持久化，不拼 prompt。
- Agent 只关心某个步骤如何生成，不直接落盘。
- Prompt 是外部文件，便于调整生成策略。
- 工作流依赖集中在 `workflow_spec.py`，GUI 和后端都围绕这份定义工作。

这种拆法的好处是后续新增步骤、替换 LLM、调整 GUI 布局、调整存储方式时，不需要把所有文件都改一遍。
