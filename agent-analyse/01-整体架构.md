# 整体架构

## 项目定位

`novel_agent_clean` 是一个面向长篇小说生产的 Agent 工作流框架。它解决的不是“单次让模型写一段文本”，而是把小说生成拆成可审查、可回滚、可继续扩展的一串阶段：

```text
项目设定
  -> 需求分析
  -> 世界观设定
  -> 角色框架
  -> 总纲
  -> 粗卷纲
  -> 完整卷纲
  -> 粗章纲
  -> 完整章节计划
  -> 章节正文
  -> 修订 / 一致性检查 / 记忆整理
```

系统的中心对象是 `NovelProject`。所有生成结果都挂在项目的 `artifacts` 字典里，所有接口、页面、任务和审计日志最终都围绕项目 ID 运转。

## 分层结构

后端代码主要分成六层：

| 层级 | 代表文件 | 职责 |
| --- | --- | --- |
| 领域层 | `domain.py` | 定义项目、产物、任务、工作流状态、接口协议和业务异常 |
| 工作流定义 | `workflow_spec.py` | 统一声明步骤顺序、依赖、页面归属和作用域 |
| 服务层 | `services.py` | 管理项目、任务、审计、Prompt、LLM 调用和 artifact 工具函数 |
| Agent 层 | `agents.py` | 根据项目和上下文渲染 prompt、调用模型、解析并校验输出 |
| 编排层 | `orchestrator.py` | 检查依赖、分派步骤、注册产物、记录审计 |
| API 层 | `api/` | 暴露 HTTP 接口、静态页面、异常处理和依赖注入 |

基础设施在 `infrastructure.py` 中实现，包括文件项目仓储、Prompt 存储、审计写入、文件任务存储、测试用内存任务存储、Mock LLM 和 OpenAI-compatible Provider。

前端代码主要在 `frontend/`：

| 文件/目录 | 职责 |
| --- | --- |
| `index.html` + `index-page.js` | 项目中心：创建、选择、编辑项目 |
| `workflow.html` + `workflow-page.js` | 主工作流页：展示完整链路和通用生成入口 |
| `outline.html` / `volume-outline.html` / `chapter-plan.html` + `structure-page.js` | 结构页：分别聚焦总纲、卷纲、章节计划 |
| `review.html` + `review-page.js` | 审查页：查看任务、产物和快照 |
| `assets/js/api-client.js` | 前端 API 适配层 |
| `assets/js/step-meta.js` | 前端步骤元数据、表单字段和接口映射 |
| `assets/js/workspace-utils.js` | 表单渲染、任务提交、字段补全、产物展示 |
| `assets/js/artifact-selectors.js` / `scope-utils.js` | 前端作用域选择和依赖状态判断 |

## 运行入口

运行入口是 `novel_agent/main.py`。它加载配置后创建 FastAPI 应用，并用 uvicorn 启动服务。

实际应用构造在 `api/app.py`：

1. 调用 `load_config()` 读取环境变量。
2. 调用 `build_container()` 组装仓储、服务、Agent 编排器和任务线程池。
3. 安装 CORS、异常处理器、静态资源和各个 router。
4. 把容器挂到 `app.state.container`，路由通过 `get_container()` 获取。

这个设计让 API 层不需要知道 LLM、文件路径、任务存储等细节，所有依赖都从容器拿。

## 核心对象关系

```text
FastAPI route
  -> AppContainer
    -> TaskService.submit(...)
      -> NovelOrchestrator.dispatch(...)
        -> ProjectService.get(...)
        -> _ensure_step_ready(...)
        -> Agent.run(...)
          -> build_generation_context(...)
          -> PromptService.render(...)
          -> LLMService.complete(...)
          -> parse generated JSON
        -> ProjectService.register_generated_artifact(...)
        -> AuditService.record(...)
```

用户看到的是“点击生成按钮”，系统内部实际发生的是“创建任务 -> 后台执行 -> 调 Agent -> 产物落盘 -> 前端轮询任务状态 -> 刷新快照”。

## 设计思路

这个项目的主要设计目标是解耦：

- 路由只处理 HTTP，不承担生成逻辑。
- 服务层只管理状态和持久化，不拼 prompt。
- Agent 只关心某个步骤如何生成，不直接落盘。
- Prompt 是外部文件，便于调整生成策略。
- 工作流依赖集中在 `workflow_spec.py`，前后端都围绕这份定义工作。

这种拆法的好处是后续新增步骤、替换 LLM、扩展前端页面、调整存储方式时，不需要把所有文件都改一遍。
