# 生成规则

所有生成类步骤（共13个）必须遵守以下规则：

## 基础原则

1. **只输出 JSON**：不输出任何解释、说明、摘要前后缀
2. **补全优于编造**：信息不足时，宁可保守，不要发明
3. **下游可消费**：每个字段都要让下游能直接使用，不能写成无意义总结
4. **不要重复**：content 不重复 structured，key_facts 不重复 constraints

## 上下文读取优先级

当多个上游产物同时存在时，按以下优先级读取：

1. **step_payload**：用户本次补充要求，最具体
2. **当前步骤的直接上游产物**：约束最直接
3. **更早期的产物**：只作背景，不得覆盖直接约束

## 禁止的废话模式

以下内容在任何字段里都禁止出现：

- "需要进一步明确"
- "后续可补充"
- "根据实际情况可能有所调整"
- "建议在后续步骤中进一步细化"
- "以上内容仅供参考"
- "有待进一步验证"

## 各步骤质量标准

### requirements
- 受众必须是具体画像，不是"广大读者"
- 边界必须是下游可直接执行的约束，不是模糊方向

### story_bible
- 世界规则必须包含代价和后果，不是装饰性描述
- 冲突必须能推动情节，不是设定罗列

### characters
- 核心角色必须有功能定位，不是标签罗列
- 角色槽位必须有生长空间，不是锁死设计

### outline
- 必须给 rough_volume_outline 提供真正可执行的蓝图
- 全书三段式必须有递进关系，不是简单划分

### rough_volume_outline
- 每卷必须有独立目标，不是平均分配
- 卷与卷之间必须有因果推进，不是机械并列

### volume_outline
- 必须拆到能进入粗章纲阶段，不是停留在总纲语言
- 卷尾钩子必须具体可承接，不是模糊悬念

### rough_chapter_plan
- 每章必须有不同推进职能，不是同质化
- main_event/conflict/hook 必须具体，不是空泛描述

### chapter_plan
- scene_summaries 必须可执行，不是抽象概括
- continuity_notes 必须是硬约束，writing_notes 必须是可执行提醒

### chapter
- 必须是小说正文，不是剧情概述、策划文案
- 结尾必须落在 hook 上，不能平收

### revision
- 必须针对节奏、表达、冲突强度做具体优化
- 不能只换同义词，不能泛泛说"优化表达"

### consistency
- 必须指出具体冲突点、证据来源、修复建议
- 不能泛泛说"整体基本一致"

### memory
- 必须抽取下游可直接拿来用的稳定事实
- 不能写成泛泛摘要文

## structured 字段设计原则

每个步骤的 structured 必须包含：

1. **本步骤的主体数据**：不是元数据，是实际产出
2. **下游必需的关键字段**：下游读这份产物需要什么字段就必须有什么字段
3. **不可省略的空字段**：缺失时用 `""`/`[]`/`{}` 填充，不要省略键

## 质量检查清单

生成完毕后自检：

- [ ] 输出的 JSON 是合法格式
- [ ] content 是正文，不是摘要
- [ ] key_facts 是事实性条目，不是感悟
- [ ] constraints 是硬约束，不是期望
- [ ] open_questions 是未决问题，不是已完成事项
- [ ] structured 包含本步骤主体数据，不是空对象
- [ ] 没有"需要进一步明确""后续可补充"等废话
- [ ] 没有重复 structured 已有的主体信息
