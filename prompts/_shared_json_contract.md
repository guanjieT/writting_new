# JSON 输出协议

所有生成步骤必须遵守以下 JSON 协议：

## 输出格式

- 只输出一个合法 JSON 对象，不要 Markdown 代码块，不要额外说明
- 不允许输出 `null` 作为任何字段的值
- 缺失字段或空值时使用：`""`（空字符串）、`[]`（空数组）、`{}`（空对象）、`false`（布尔）

## 顶层字段职责

| 字段 | 职责 | 禁止 |
|------|------|------|
| `content` | 纯文本正文，允许自然分段 | 禁止放对象、数组、标题列表、条目罗列 |
| `overview` | 一两句话说明本步骤产物的叙事作用 | 禁止写成章节目录、要点列表 |
| `key_facts` | 事实性条目，可被下游直接继承使用 | 禁止写成感悟、总结、分析 |
| `constraints` | 已确立的后续不能随意改动的边界 | 禁止写成建议、愿望、期待 |
| `open_questions` | 本步骤结束后仍悬而未决、需要后续承接的问题 | 禁止写成已完成事项 |
| `best_for_reason` | 一句话说明这份产物为什么适合给下游使用 | 禁止写成自我评价、泛泛评价 |
| `structured` | 结构化数据主体，必须包含本步骤核心字段 | 禁止空对象作为主体数据 |

## 各字段写法要求

### content
- 纯文本，可分段
- 不得包含对象、数组、列表
- 不得使用 Markdown 标题、列表语法
- 是本步骤产物的正文内容，不是摘要

### overview
- 一两句话
- 说明"这份产物完成了什么叙事/结构功能"
- 下游能直接判断这份产物是否有用

### key_facts
- 数组，每条是一个事实性陈述
- 事实性 = 可作为下游约束的事实，不是观点、感悟、分析
- 长度：一条Facts建议不超过50字
- 数量：3~10条为宜，聚焦，不要堆砌

### constraints
- 数组，每条是一个硬约束
- 下游必须遵守，不能自行覆盖
- 来自本章正文确立的设定、人物关系、世界规则、事件状态

### open_questions
- 数组，每条是一个悬而未决的问题
- 下游需要承接或回答
- 不是已完成的事项

### best_for_reason
- 一句话，说清楚"这份内容适合谁来用、用来做什么"
- 理由要具体，不要泛泛

### structured
- 对象，包含本步骤的核心结构化字段
- 必须是本步骤产出的主体数据
- 各步骤有自己的结构化字段要求（见各 prompt）

## 禁止行为

1. 禁止输出任何解释性文字在 JSON 之外
2. 禁止把 `content` 写成摘要、把 `structured` 当附庸
3. 禁止在 `key_facts` 里写感悟、总结、分析
4. 禁止在 `constraints` 里写期望、建议、愿望
5. 禁止在 `open_questions` 里写已完成事项
6. 禁止用 `null` 表示空值
7. 禁止在 `content` 里使用条目、列表、标题
8. 禁止在 `structured` 里空着核心字段不填
